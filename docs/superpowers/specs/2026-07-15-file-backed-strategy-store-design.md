# 文件化网格策略存储设计

## 背景

网格策略生成器当前将左侧“已保存策略”写入浏览器 `localStorage`。浏览器按协议、主机名和端口隔离存储，因此 `localhost:52341`、`localhost:55018` 与 `127.0.0.1:18765` 无法共享策略。端口或主机名变化后，旧策略仍在原浏览器空间中，但新页面无法加载。

本次改造将项目内文件作为唯一正式策略库，并提供一次性历史端口迁移。CSV 导出继续作为可用 Excel 打开的单策略副本。

## 目标

- 页面在任何本地服务端口下都读取同一个策略库。
- 保存、覆盖和删除策略后，刷新页面仍能恢复左侧列表。
- 统一策略库纳入 Git 跟踪，但页面不自动执行 Git 命令。
- 从 `localhost:52341` 和 `localhost:55018` 迁移旧 `localStorage` 记录。
- 迁移不删除旧浏览器数据，并且可以安全重复执行。
- 保持现有网格计算、留利润、估值快照和 CSV 导出行为。

## 非目标

- 不自动执行 `git add`、`git commit` 或 `git push`。
- 不将策略同步到云端或局域网服务。
- 不把 CSV 作为页面的正式策略数据库。
- 不删除历史地址中的 `localStorage`。
- 不修改网格计算、估值或留利润算法。

## 正式策略库

正式文件为：

`grid-strategy-generator/data/saved-strategies.json`

文件纳入 Git 跟踪，仓库初始内容为空策略库：

```json
{
  "version": 2,
  "records": []
}
```

文件沿用前端现有 version 2 保存结构。每条记录包含标的代码和名称、参数、保存时间以及可选估值快照。页面不直接写文件，而是通过本地 Python 服务读写。

写入流程：

1. 服务验证完整保存结构。
2. 在策略文件同目录创建临时文件。
3. 写入 UTF-8 JSON 并刷新内容。
4. 原子替换正式文件。

服务使用进程内锁串行化读取、写入与迁移，避免多个本地请求交叉覆盖。写入失败时保留原文件。

## 本地接口

接口仅由绑定在 `127.0.0.1` 的本地服务提供。

### `GET /api/strategies`

读取并返回完整策略库。如果文件不存在，返回空 version 2 结构。文件存在但无法解析或校验失败时返回明确错误，不自动覆盖损坏文件。

### `PUT /api/strategies`

接收页面当前的完整 version 2 策略列表，用于新增、覆盖和删除。请求体设置合理大小上限，并校验版本、记录结构、参数、保存时间和可选估值快照。校验成功后原子写入。

### `POST /api/strategies/import`

接收历史端口提交的标准 version 2 保存结构。迁移页面先使用现有前端 `parseStore` 将 version 1 或 version 2 的旧 `localStorage` 规范化，再提交给服务。服务复用同一校验规则，跳过损坏记录，将有效记录与正式策略库合并。同一记录键冲突时保留 `savedAt` 较新的记录，并返回导入、更新、跳过和总记录数量。

### `GET /api/strategies/migration-status`

返回历史端口的可用状态。迁移按钮只访问已成功启动辅助监听的端口，并在最终结果中列出无法监听的端口。

## 页面数据流

页面启动后：

1. 调用 `GET /api/strategies`。
2. 使用现有前端保存结构解析器再次校验响应。
3. 渲染左侧策略列表。

用户保存策略时：

1. 继续生成并下载 CSV。
2. 在内存列表中按现有记录键新增或覆盖。
3. 调用 `PUT /api/strategies` 持久化完整列表。
4. 只有文件写入成功后才确认左侧列表保存成功。

用户删除策略时，页面先构造删除后的列表，再调用 `PUT /api/strategies`。写入成功才更新界面；失败则保留原列表。

JSON 写入失败不阻止 CSV 下载。页面分别报告“CSV 已下载”和“策略文件保存失败”，避免把导出成功误认为正式保存成功。

`localStorage` 不再承担正常保存和加载，只在历史地址迁移页面中读取。

## 历史端口迁移

正常服务除主端口外，尝试在 `127.0.0.1:52341` 和 `127.0.0.1:55018` 启动辅助迁移监听。辅助监听使用同一个策略文件存储实例。

左侧“已保存策略”区域增加“迁移旧策略”按钮。用户点击后，当前浏览器依次访问：

1. `http://localhost:52341/?migrate=1`
2. `http://localhost:55018/?migrate=1`
3. 返回原主页面

开始迁移前，主页面调用 `GET /api/strategies/migration-status`，只把可用端口加入访问序列。迁移页面运行在历史地址本身，因此能够读取该地址对应的 `localStorage`。页面读取键 `grid-strategy-generator.saved-strategies.v1`，使用现有 `parseStore` 规范化数据，向同源的 `POST /api/strategies/import` 提交，然后进入下一个可用地址。流程完成后返回主页面并重新调用 `GET /api/strategies`。

迁移满足以下规则：

- 不删除或修改旧 `localStorage`。
- 没有旧记录时按成功处理，导入数量为零。
- 重复执行时按记录键和保存时间去重，结果不重复增长。
- 某个历史端口不可用时报告该端口失败，并允许继续处理另一个端口。
- 辅助迁移端口只服务本机请求，不扩大网络暴露范围。

## 错误处理

- 策略文件缺失：返回空库，首次保存时创建目录和文件。
- 策略文件损坏：返回 `STRATEGY_STORE_CORRUPT`，不覆盖文件。
- 请求结构无效：返回 `INVALID_STRATEGY_STORE`，不写入文件。
- 请求过大：返回 `REQUEST_TOO_LARGE`。
- 文件写入失败：返回 `STRATEGY_STORE_WRITE_FAILED`，保留原文件。
- 历史端口绑定失败：主工具仍可启动，页面显示具体不可用端口。
- 网络或接口失败：左侧保留最后一次已确认的列表，显示可重试错误。

公开错误响应不包含 Python 堆栈或本地敏感路径。

## 文件边界

- `grid-strategy-generator/strategy_file_store.py`：策略结构校验、读取、原子写入和迁移合并。
- `grid-strategy-generator/server.py`：策略 API、请求大小限制、主服务与历史迁移监听。
- `grid-strategy-generator/index.html`：异步加载、保存、删除、迁移按钮和状态显示。
- `grid-strategy-generator/data/saved-strategies.json`：Git 跟踪的正式策略库。
- `grid-strategy-generator/tests/test_strategy_file_store.py`：文件存储与合并规则测试。
- `grid-strategy-generator/tests/test_server.py`：策略接口与历史端口监听测试。
- `grid-strategy-generator/tests/grid-calculator.test.mjs`：前端 API 数据流、错误状态和迁移流程测试。

不拆分现有单文件前端，避免与本次目标无关的重构。

## 测试与验收

自动测试覆盖：

- 空文件初始化、有效文件读取和损坏文件拒绝。
- 原子写入失败时旧文件不变。
- 前端将 version 1/2 旧记录规范化为 version 2，并跳过损坏记录。
- 迁移新增、更新、重复执行、旧记录不覆盖新记录。
- `GET`、`PUT`、`POST import` 的成功与公开错误响应。
- 历史端口部分不可用时主服务正常运行。
- 页面启动从 API 加载，不再从当前端口 `localStorage` 加载。
- 保存和删除只在服务器确认后更新界面。
- JSON 保存失败时 CSV 仍可下载并显示分别结果。
- 迁移按钮按 `52341`、`55018` 顺序执行并刷新正式列表。
- 现有 Python 和 Node 测试全部继续通过。

浏览器验收：

1. 保存一条策略并刷新，左侧仍显示该策略。
2. 改用另一个可用主端口启动，仍读取同一策略。
3. 在历史地址准备旧 `localStorage`，执行迁移后正式文件出现记录。
4. 重复迁移不产生重复记录。
5. 删除策略后刷新，记录保持删除状态。
6. 模拟策略文件不可写，CSV 仍下载且页面明确报告文件保存失败。

## Git 行为

`data/saved-strategies.json` 是普通受跟踪文件。页面修改它之后，Git 工作区会显示变更；工具不暂存、不提交也不推送该文件，所有 Git 操作由用户决定。
